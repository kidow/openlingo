# OpenLingo 프로젝트 구현 계획서

부제: 언어별 손글씨 연습 MVP에서 단어·문장 단위 필기 연습으로 확장하는 학습 도구

---

## 1. 프로젝트 개요

### 1.1 프로젝트 목적

**OpenLingo**는 태블릿 환경에서 다양한 언어의 문자를 직접 따라 쓰며 연습할 수 있는 손글씨 학습 도구다. 초기에는 **한국어 문자 단위 손글씨 연습 MVP**로 시작하고, 이후에는 **다국어 선택형 단어 및 문장 단위 필기 연습**까지 확장한다.

사용자는 아이패드와 Apple Pencil 조합을 중심으로, 화면 위의 예시 가이드를 따라 직접 글씨를 쓰며 반복 학습할 수 있어야 한다. 또한 동일한 구조를 바탕으로 **갤럭시 탭 + S Pen** 등 펜 지원 태블릿에서도 핵심 경험을 제공하는 것을 목표로 한다.

### 1.2 핵심 가치

- 언어 학습에 필요한 **실제 필기 훈련** 제공
- 실제 손글씨 연습장처럼 **격자 칸 위 예시 문자를 덧대어 따라 쓰는** 직관적 UX
- 손글씨와 예시 간 **유사도 백분율** 제공
- 태블릿 펜 환경에 최적화된 학습 흐름
- 문자 단위에서 문장 단위까지 자연스럽게 확장 가능한 구조

### 1.3 제품 형태

- 1차 목표: 개인용 태블릿 손글씨 학습 웹앱
- 핵심 사용 환경: iPad + Apple Pencil
- 확장 고려 환경: Galaxy Tab + S Pen, 기타 펜 지원 태블릿
- 실행 방식: 브라우저 실행 및 홈 화면 설치형 웹앱 경험 제공

### 1.4 제품 비전

OpenLingo는 단순한 따라쓰기 앱이 아니라, **문자 형태를 몸으로 익히는 언어 필기 연습 도구**를 지향한다.

제품은 아래 단계로 진화한다.

1. **문자 단위 연습**
2. **반복 셀 기반 문자 워크시트**
3. **단어 단위 연습**
4. **문장 단위 연습**
5. **언어별 필기 규칙을 반영한 고급 피드백**

### 1.5 언어 지원 전략

- 초기 MVP는 **한국어 전용**으로 시작한다.
- 각 언어는 가장 기본이 되는 문자군부터 시작하며, 한국어 MVP는 `ㄱ`, `ㄴ`, `ㄷ` 같은 자음·모음 기초 연습부터 구현한다.
- 이후 확장 언어는 영어, 일본어, 중국어, 러시아어, 포르투갈어, 이탈리아어, 스페인어, 독일어, 프랑스어, 아랍어를 우선 고려한다.
- 제품 구조는 처음부터 **언어 선택 후 학습 시작** 흐름을 전제로 설계한다.

## 2. 문제 정의 및 제품 방향

### 2.1 해결하려는 문제

외국어를 배울 때 문자를 눈으로 익히는 것만으로는 충분하지 않다. 특히 다음 문제가 존재한다.

- 문자를 실제로 반복해서 써보는 훈련이 필요하다.
- 종이 프린트나 PDF는 저장, 반복, 피드백 측면에서 비효율적이다.
- 태블릿 펜 환경에서 바로 열고 연습할 수 있는 도구가 부족하다.
- “내가 제대로 썼는지”를 빠르게 가늠할 수 있는 보조 지표가 필요하다.

### 2.2 제품 방향

초기 제품은 “완벽한 필적 판정기”가 아니라 아래 기능에 집중한다.

- 예시를 따라 쓰기 쉽게 만든다.
- 펜 입력을 자연스럽게 기록한다.
- 예시와의 **형태 유사도**를 백분율로 제공한다.
- 같은 문자를 반복 연습하고 개선 과정을 확인하게 한다.

### 2.3 평가 철학

OpenLingo의 점수는 절대적 정답 판정이 아니라 **학습 보조 지표**다.

- 초기 점수는 “형태 유사도” 중심이다.
- 폰트를 얼마나 복제했는지보다 “비슷하게 썼는가”를 본다.
- 초기 버전에서는 획순, 필압, 고급 필체 평가는 보류한다.
- 후속 버전에서 언어별 규칙과 stroke-aware 비교를 도입한다.

## 3. 범위 정의

### 3.1 MVP 범위

MVP에서 반드시 구현할 기능은 다음과 같다.

#### 문자 연습

- 문자 단위 연습 화면 제공
- 각 언어별로 가장 기본 문자부터 학습할 수 있어야 하며, 한국어 MVP는 `ㄱ`, `ㄴ`, `ㄷ` 등 기초 자음·모음부터 제공
- 각 문자마다 예시 SVG 또는 중심선 가이드가 들어간 **정사각형 격자 칸** 표시
- 사용자가 그 위에 펜으로 덧대어 쓰기
- 현재 글자 초기화 / 다시 쓰기 / 마지막 획 되돌리기
- 입력이 멈추면 debounce 기반으로 현재 글자 유사도 자동 계산

#### 입력 처리

- 펜 기반 입력 수집
- 손가락 입력 허용 여부 설정 가능
- stroke 단위 저장
- 입력 종료 판단을 위한 debounce 유틸 적용

#### 렌더링

- 예시 레이어
- 사용자 손글씨 레이어
- 정사각형 격자 칸, 줄, 가이드 표시

#### 비교/평가

- 예시 경로와 사용자 입력 비교
- 0~100 정수 백분율 반환
- 간단한 피드백 문구 제공

#### 로컬 저장

- 최근 연습 기록 저장
- 문자별 최고 점수 또는 최근 점수 저장
- 오프라인 사용 가능

#### 설치형 웹앱 경험

- 홈 화면 설치 가능
- 오프라인 기본 동작
- 정적 자산 캐싱

### 3.2 MVP 제외 범위

초기에는 다음 기능을 제외한다.

- 한국어 외 실서비스 언어 학습 콘텐츠
- 계정 시스템
- 서버 동기화
- 실시간 협업
- AI 필적 교정
- ML 기반 인식 모델
- 네이티브 앱 배포
- 문장 생성형 학습 추천 시스템

### 3.3 확장 범위

후속 버전에서 고려할 기능은 다음과 같다.

- 영어, 일본어, 중국어, 러시아어, 포르투갈어, 이탈리아어, 스페인어, 독일어, 프랑스어, 아랍어 지원
- 단어 단위 연습
- 문장 단위 연습
- 여러 칸을 가진 워크시트 모드
- 오답 문자 반복 모드
- 언어별 세트 패키지
- 획순 애니메이션
- 획순 평가
- 필기 리플레이
- 결과 내보내기
- 연습지 PDF 출력

## 4. 타겟 디바이스 및 플랫폼 전략

### 4.1 핵심 타겟

초기 최적화 대상은 다음이다.

- iPad
- Apple Pencil
- Safari 또는 홈 화면 설치형 웹앱 실행

### 4.2 확장 타겟

동시에 아래 환경도 고려한다.

- Galaxy Tab
- S Pen
- Chrome 또는 Samsung Internet

### 4.3 플랫폼 전략

핵심 전략은 다음과 같다.

- **iPad 중심으로 UX를 먼저 다듬는다.**
- 그러나 구조는 **펜 지원 태블릿 전반**에서 동작하도록 설계한다.
- “iPad 전용 앱”으로 제한하지 않는다.
- 제품 설명은 아래처럼 정의한다.

> 아이패드에 최적화해 시작하되, 갤럭시 탭 등 펜 지원 태블릿에서도 사용 가능한 손글씨 학습 도구

### 4.4 플랫폼 차이에 대한 현실적 가정

기능 수준에서는 아래 경험이 대부분 공통으로 가능하다.

- 홈 화면 설치
- 오프라인 사용
- 펜 입력
- 손글씨 렌더링
- 예시 오버레이
- 유사도 계산

다만 세부 감각은 기기와 브라우저 조합에 따라 달라질 수 있다.

- 입력 지연감
- 필압/기울기 세부 지원
- 팜 리젝션 체감
- 설치 후 앱처럼 느껴지는 정도

따라서 목표는 “완전히 동일한 UX”가 아니라 **핵심 경험의 일관성 확보**다.

## 5. 사용자 시나리오

### 5.1 문자 단위 연습 시나리오

1. 사용자가 OpenLingo를 연다.
2. 학습 언어를 선택한다.
3. 문자 목록에서 특정 문자를 선택한다.
4. 화면에 해당 문자 예시와 연습 칸이 나타난다.
5. 사용자는 펜으로 예시 위에 직접 글씨를 쓴다.
6. 앱은 stroke 데이터를 저장하고 화면에 렌더링한다.
7. 입력이 멈추면 debounce가 동작해 유사도 %를 자동 계산한다.
8. 결과를 보고 초기화하거나 다시 쓰거나 다음 문자로 이동한다.

### 5.2 반복 훈련 시나리오

- 한 문자를 여러 번 연습한다.
- 시도마다 점수를 확인한다.
- 최고 점수와 최근 점수를 비교한다.

### 5.3 향후 단어·문장 확장 시나리오

- 사용자가 문장 모드를 선택한다.
- 단어 또는 짧은 문장이 여러 칸에 표시된다.
- 각 단위별 점수 또는 전체 평균 점수를 확인한다.
- 반복적으로 문장을 따라 쓴다.

## 6. 제품 요구사항

### 6.1 기능 요구사항

#### FR-01 연습 화면

- 문자 또는 텍스트를 연습 대상으로 표시해야 한다.
- 초기 MVP에서는 한국어 문자만 제공해야 한다.
- 향후 확장을 위해 언어 선택 진입점을 제공해야 한다.
- 예시 SVG 가이드를 제공해야 한다.
- 실제 손글씨 연습장처럼 정사각형 격자 칸 안에 예시 문자를 배치해야 한다.
- 기준선, 칸, 격자 등 보조 가이드를 옵션으로 제공해야 한다.

#### FR-02 펜 입력

- Pointer Events 기반 입력을 처리해야 한다.
- `pointerdown`, `pointermove`, `pointerup`, `pointercancel` 흐름을 지원해야 한다.
- stroke는 점 배열 형태로 저장되어야 한다.

#### FR-03 드로잉 엔진

- 입력 중 실시간으로 필기를 렌더링해야 한다.
- 선 굵기, 색상, smoothing 정도를 조정 가능해야 한다.
- 펜 환경에서 자연스럽게 보여야 한다.

#### FR-04 편집 동작

- 현재 글자 초기화
- 마지막 획 되돌리기
- 다시 쓰기
- 다음 문자로 이동

#### FR-05 유사도 평가

- 정답 예시와 사용자 입력을 비교해야 한다.
- 결과는 0~100 정수 백분율이어야 한다.
- 비교 전에 위치/크기 정규화가 수행되어야 한다.
- 별도 채점 버튼 없이 debounce 유틸을 사용해 입력 종료 시 자동 평가해야 한다.

#### FR-06 결과 표시

- 점수 표시
- 간단한 평가 문구 표시
- 최고 점수 갱신 여부 표시 가능

#### FR-07 로컬 저장

- 문자별 시도 기록 저장
- 최근 학습 상태 저장
- 설정값 저장

#### FR-08 설치형 웹앱 경험

- 홈 화면 설치 가능
- 오프라인 핵심 기능 보장
- 앱 아이콘 및 manifest 구성

### 6.2 비기능 요구사항

#### NFR-01 성능

- 필기 입력 지연이 체감되지 않아야 한다.
- 문자 단위 비교 연산은 빠르게 완료되어야 한다.

#### NFR-02 안정성

- 오프라인에서도 핵심 기능이 동작해야 한다.
- 새로고침이나 재실행 후 상태가 망가지지 않아야 한다.

#### NFR-03 사용성

- 태블릿 화면에서 조작이 쉬워야 한다.
- 연습 영역이 충분히 넓어야 한다.
- 의도치 않은 스크롤/줌을 최소화해야 한다.
- 예시 글자와 사용자 필기가 같은 격자 칸 안에서 자연스럽게 겹쳐 보여야 한다.

#### NFR-04 확장성

- 언어별 문자 세트 추가가 쉬워야 한다.
- 문장 모드 확장이 자연스러워야 한다.
- 비교 엔진을 단계적으로 개선할 수 있어야 한다.

## 7. 기술 전략

### 7.1 플랫폼 선택

OpenLingo는 설치형 웹앱 경험을 제공하는 웹 프로젝트로 구현한다.

선택 이유:

- 무료 개발 및 사용 가능
- 빠른 프로토타이핑 가능
- 펜 입력, 캔버스 드로잉, 오프라인 저장 구현 가능
- 개인용 학습 도구로 충분한 사용성 확보 가능

### 7.2 프론트엔드 스택 제안

#### 권장 스택

- React
- TypeScript
- Vite 또는 Next.js
- Tailwind CSS
- shadcn/ui
- Canvas + SVG 혼합 렌더링
- IndexedDB
- Service Worker

#### 현실적 추천

현재 목적이 단일 제품 MVP에 가깝기 때문에 다음 구성이 가장 합리적이다.

- **Vite + React + TypeScript + Tailwind CSS + shadcn/ui**

이유:

- 가볍다.
- 개발 속도가 빠르다.
- 설치형 웹앱 구성도 단순하다.
- 개인용 도구에 적합하다.
- Tailwind CSS로 태블릿 최적화 UI와 상태별 스타일을 빠르게 조정할 수 있다.
- shadcn/ui를 기반으로 공통 UI 품질과 접근성을 빠르게 확보할 수 있다.

#### UI 프레임워크 원칙

- 기본 UI 프레임워크는 **shadcn/ui**로 고정한다.
- 버튼, 시트, 다이얼로그, 폼, 토스트, 드롭다운 등 범용 UI는 shadcn/ui 컴포넌트를 우선 사용한다.
- OpenLingo 전용 화면은 shadcn/ui primitive 위에 제품 전용 컴포넌트를 조합해 만든다.
- 다른 대형 UI 프레임워크를 병행 도입하지 않는다. 디자인 일관성과 유지보수 비용을 우선한다.
- 공통 컴포넌트는 `src/components/ui` 아래에 두고, 제품 전용 화면 구성은 `src/features` 또는 개별 도메인 컴포넌트로 분리한다.

#### MCP 활용 원칙

- shadcn/ui 관련 구현을 빠르게 진행해야 할 때는 **shadcn MCP**를 우선 사용한다.
- Codex에서 반복적으로 컴포넌트를 찾거나 registry 기반 추가가 필요할 때 MCP 사용을 허용한다.
- shadcn/ui MCP 관련 기준 문서는 [shadcn/ui MCP for Codex](https://ui.shadcn.com/docs/mcp#codex)를 따른다.
- MCP는 생산성 도구로 사용하되, 최종 반영 전에는 프로젝트 구조, Tailwind 설정, 접근성, 번들 크기 영향을 직접 검토한다.

### 7.3 렌더링 전략

권장 구조는 **예시 SVG + 사용자 Canvas** 의 이중 레이어 방식이다.

#### 예시 레이어

- SVG 기반 템플릿
- 낮은 opacity
- 가이드 역할

#### 사용자 레이어

- Canvas 기반 필기
- 실시간 입력 렌더링
- 성능상 유리

이 구조가 좋은 이유:

- 예시는 SVG로 관리하기 쉽다.
- 필기는 Canvas가 자연스럽다.
- 비교 시에는 사용자 입력을 point/path 데이터로 변환할 수 있다.

## 8. 핵심 기술 설계

### 8.1 입력 처리 설계

#### 입력 이벤트

- `pointerdown`
- `pointermove`
- `pointerup`
- `pointercancel`

#### 포인트 데이터 구조

```ts
export type StrokePoint = {
  x: number;
  y: number;
  t: number;
  pressure?: number;
};

export type Stroke = {
  id: string;
  points: StrokePoint[];
};
```

#### 처리 원칙

- 한 번 누르고 떼기까지를 1 stroke로 본다.
- 좌표는 연습 영역 기준 로컬 좌표로 저장한다.
- 필요하면 `pointerType`으로 pen/touch를 구분한다.
- 손가락 입력 차단 옵션을 둘 수 있다.
- 마지막 입력 이후 일정 시간 변화가 없으면 debounce로 평가를 트리거한다.

### 8.2 연습 시도 데이터 구조

```ts
export type PracticeAttempt = {
  contentId: string;
  language: string;
  mode: "character" | "word" | "sentence";
  strokes: Stroke[];
  createdAt: number;
  similarity?: number;
};
```

이 구조는 문자 연습뿐 아니라 이후 단어/문장 모드에도 그대로 확장 가능하다.

### 8.3 템플릿 데이터 구조

```ts
export type WritingTemplate = {
  id: string;
  language: string;
  mode: "character" | "word" | "sentence";
  label: string;
  viewBox: [number, number, number, number];
  guidePathD: string;
  strokePaths?: string[];
  difficulty?: number;
};
```

#### 설계 의도

- 현재는 문자 중심으로 사용
- 이후 단어/문장까지 같은 구조로 확장
- 획 단위 비교가 필요하면 `strokePaths`를 활용

### 8.4 SVG 비교 원칙

비교 대상은 SVG 문자열 그 자체가 아니라 **geometry/path** 다.

즉, 아래 기준으로 비교한다.

- path shape
- resampled points
- normalized bounds

### 8.5 유사도 엔진 MVP 설계

초기 비교 방식은 다음 절차를 따른다.

1. 정답 SVG path를 일정 개수 점으로 샘플링
2. 사용자 stroke를 하나의 polyline 또는 path로 변환
3. 사용자 입력도 동일 개수로 재샘플링
4. 양쪽을 bounding box 기준으로 정규화
5. 대응 점 간 평균 거리 계산
6. 거리값을 0~100 백분율 점수로 변환

#### 점수 의미 예시

- 90~100: 매우 유사
- 75~89: 좋음
- 50~74: 연습 필요
- 0~49: 많이 다름

### 8.6 비교 엔진 고도화 로드맵

#### 단계 1

- 점열 기반 비교

#### 단계 2

- 역방향 path 허용

#### 단계 3

- 회전 소폭 보정

#### 단계 4

- stroke-aware 비교
- 획 개수, 시작점, 방향, 길이 반영

#### 단계 5

- 언어별 규칙 반영
- 한글: 자모 배치
- 일본어/중국어: 획순 비중 확대
- 라틴 필기체: baseline/slant 반영

## 9. UI/UX 설계

### 9.1 핵심 화면 구성

#### 홈 화면

- 언어 선택
- 연습 모드 선택
- 최근 연습 이어하기

- 격자 칸 안의 예시 SVG 가이드
- 예시와 사용자 입력이 겹쳐지는 필기 영역
- 점수 패널

  - 초기화
  - 되돌리기
  - 다시 쓰기
  - 다음

#### 향후 문장 연습 화면

- 단어 또는 문장 표시
- 여러 연습 셀 또는 긴 필기 영역
- 구간별 또는 전체 점수 제공

### 9.2 태블릿 최적화 포인트

- 툴바는 큰 터치 타겟 사용
- 필기 영역은 넓게 확보
- 툴바와 필기 영역이 명확히 분리되도록 구성
- 손바닥 오입력에 덜 민감하게 설계

### 9.3 스타일링 및 인터랙션 체크 포인트

- `touch-action: none`
- `user-select: none`
- 불필요한 드래그/스크롤 방지
- 브라우저 UI와 충돌 최소화

#### 애니메이션/고급 UI 도입 원칙

- 애니메이션은 학습 흐름을 방해하지 않는 범위에서만 사용한다.
- 점수 갱신, 다음 문제 전환, 패널 진입/이탈, 설치 유도 등 **의미가 있는 피드백**에만 애니메이션을 쓴다.
- 기본 우선순위는 **shadcn/ui + Tailwind**로 정적 구조를 먼저 완성하고, 필요할 때만 보조 라이브러리를 붙인다.
- 애니메이션 소스 탐색이 필요하면 아래 MCP를 우선순위대로 검토한다.
- 1순위: [Magic UI MCP](https://magicui.design/docs/mcp)
- 2순위: [UI Layouts MCP](https://www.npmjs.com/package/@ui-layouts/mcp)
- 3순위: [React Bits MCP](https://www.reactbits.dev/get-started/mcp)
- Magic UI는 shadcn/ui와 조합이 쉬운 편이므로, OpenLingo의 보조 애니메이션과 마케팅성 UI에 가장 먼저 검토한다.
- React Bits와 UI Layouts는 필요한 조각만 선택적으로 가져오고, 라이선스/호환성/스타일 일관성을 확인한 뒤 반영한다.
- 연습 캔버스 자체에는 과한 모션을 넣지 않는다. 필기 경험의 안정성과 응답성이 우선이다.

### 9.4 연습 레이아웃 방향

첨부 이미지와 유사한 방향을 따른다.

- 문자마다 실제 공책 같은 정사각형 격자 셀 제공
- 예시는 희미하게 깔고 그 위에 직접 덧대어 씀
- 반복 연습용 셀을 추후 추가
- 초기 MVP는 “한 번에 한 문자 크게 연습”하는 방식이 가장 적합

## 16. 품질 기준

### 16.1 완료 정의

- 문자 예시가 보인다.
- 펜으로 자연스럽게 필기할 수 있다.
- 유사도 %가 계산된다.
- 되돌리기/지우기가 된다.
- 새로고침 후 상태 일부가 유지된다.
- 홈 화면 설치 후 재실행할 수 있다.

### 16.2 수용 기준 예시

#### AC-01 입력

- 펜 입력 시 선이 자연스럽게 이어져야 한다.
- stroke 종료 후 점 배열로 저장되어야 한다.

#### AC-02 점수

- 매우 유사한 입력은 높은 점수를 반환해야 한다.
- 전혀 다른 입력은 낮은 점수를 반환해야 한다.
- 반환값은 0~100 정수여야 한다.

#### AC-03 오프라인

- 네트워크 없이도 최근 사용 템플릿 기준 연습이 가능해야 한다.

#### AC-04 플랫폼

- iPad 중심으로 안정 동작해야 한다.
- Galaxy Tab에서도 핵심 기능은 동작해야 한다.

## 22. 구현 원칙 요약

- 제품명은 **OpenLingo**로 고정한다.
- 초기 목표는 **언어별 손글씨 문자 연습 MVP**다.
- 이후 **단어와 문장 연습**으로 확장한다.
- 기본 UI 프레임워크는 **shadcn/ui**로 통일한다.
- 공통 UI는 shadcn/ui를 우선하고, 제품 고유 경험은 그 위에 조합해 만든다.
- 구현 중 필요 시 shadcn MCP를 우선 활용한다.
- 애니메이션은 Magic UI / UI Layouts / React Bits MCP를 참고하되, 학습 흐름을 해치지 않는 범위에서만 도입한다.
- 비교 대상은 SVG 문자열이 아니라 **geometry/path** 다.
- 초기 점수는 **형태 유사도 중심의 학습 보조 지표**다.
- iPad 최적화로 시작하되, 펜 지원 태블릿까지 고려한다.

## 23. 부록: 유사도 엔진 MVP 개념 요약

### 목표

예시 경로와 사용자 손글씨 입력의 형태 유사도를 **0~100 정수 백분율**로 반환한다.

### 알고리즘 개요

1. 정답 path 샘플링
2. 사용자 입력 resample
3. 위치/크기 정규화
4. 평균 거리 계산
5. 거리값을 점수로 변환

### 주의사항

- 정답 템플릿은 중심선 형태가 유리하다.
- outline 폰트 path는 초기 품질이 떨어질 수 있다.
- 점수는 참고값으로 사용한다.

OpenLingo는 태블릿 펜 환경에서 충분히 구현 가능한 손글씨 학습 프로젝트다. 초기에는 문자 단위 연습과 유사도 계산에 집중하고, 이후 단어와 문장 단위 필기 연습으로 확장하는 것이 가장 현실적이다. iPad와 Apple Pencil을 중심으로 시작하되, Galaxy Tab과 S Pen 환경까지 고려하는 구조로 설계하면 장기적으로 더 유연한 제품이 된다.
